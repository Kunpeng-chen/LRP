# LRP Frame Format

This document defines the packet and frame structure for LRP.

The LRP frame format is designed for:

1. Small payload overhead.
2. Easy MCU parsing.
3. Deterministic memory usage.
4. Compatibility with low-power LoRa links.
5. Future protocol extensibility.

The MVP uses a compact fixed header followed by payload and integrity fields.

---

# 1. Packet Structure

An LRP packet is composed of:

```text
+----------------+
| Fixed Header   |
+----------------+
| Payload        |
+----------------+
| Integrity      |
+----------------+
```

Recommended packet flow:

```text
Application Payload
    ↓
LRP Packet Builder
    ↓
Optional Encryption
    ↓
CRC / MIC
    ↓
LoRa PHY Payload
```

---

# 2. MVP Fixed Header

Recommended MVP header:

| Field | Size | Description |
| --- | --- | --- |
| Version | 1 byte | Protocol version |
| Flags | 1 byte | ACK, encryption, relay flags |
| PacketType | 1 byte | DATA, ACK, CONTROL, etc |
| TTL | 1 byte | Remaining relay hops |
| NetworkID | 2 bytes | Logical network identifier |
| SrcID | 2 bytes | Source node |
| DstID | 2 bytes | Destination node |
| Seq | 2 bytes | Sequence number |
| PayloadLength | 1 byte | Payload size |

Total MVP fixed header size:

```text
13 bytes
```

This is small enough for constrained LoRa payload budgets.

---

# 3. Binary Layout

Recommended binary layout:

```text
Byte 0   : Version
Byte 1   : Flags
Byte 2   : PacketType
Byte 3   : TTL
Byte 4-5 : NetworkID
Byte 6-7 : SrcID
Byte 8-9 : DstID
Byte 10-11 : Seq
Byte 12  : PayloadLength
Byte 13..N : Payload
Byte N+1..N+2 : CRC16
Optional trailing MIC
```

All multi-byte fields should use little-endian encoding unless otherwise specified.

---

# 4. Flags Field

The Flags field controls optional packet behavior.

Suggested MVP flag layout:

| Bit | Meaning |
| --- | --- |
| bit0 | ACK required |
| bit1 | Packet is ACK |
| bit2 | Encrypted payload |
| bit3 | MIC present |
| bit4 | Broadcast packet |
| bit5 | Relay-generated |
| bit6 | Reserved |
| bit7 | Reserved |

Example:

```text
0b00000001
```

means:

```text
ACK required
```

---

# 5. Packet Types

Suggested packet type values:

| Type | Value |
| --- | --- |
| DATA | 0x01 |
| ACK | 0x02 |
| HELLO | 0x03 |
| CONTROL | 0x04 |
| DIAG | 0x05 |
| ROUTE_HINT | 0x06 |

Only DATA and ACK are mandatory for MVP.

---

# 6. Sequence Numbers

The sequence number is generated by the source node.

Requirements:

1. Increment monotonically.
2. Wraparound allowed.
3. Combined with SrcID to detect duplicates.
4. Reused sequence numbers should be separated by cache expiration time.

Recommended size:

```text
16-bit
```

for MVP.

Future versions may support 32-bit sequence numbers.

---

# 7. TTL Semantics

TTL controls forwarding depth.

Rules:

1. Endpoint sets initial TTL.
2. Relay decrements TTL before forwarding.
3. Relay must drop packet when TTL reaches zero.
4. Base station may accept TTL zero packets addressed to itself.

Example:

```text
TTL=2
```

allows:

```text
source → relay1 → relay2 → destination
```

---

# 8. Payload Section

The payload contains application data.

Payload guidelines:

| Payload Type | Recommendation |
| --- | --- |
| Telemetry | Binary packed fields |
| Sensor data | Fixed-size structures |
| Control commands | Small compact TLV |
| Diagnostics | Optional text or binary |

Avoid:

- JSON
- XML
- Verbose ASCII protocols

because LoRa airtime is expensive.

Recommended payload size:

| Scenario | Size |
| --- | --- |
| Typical telemetry | 8–32 bytes |
| Large telemetry | 32–64 bytes |
| Absolute upper bound | Depends on LoRa PHY |

---

# 9. CRC16

Every MVP packet should include CRC16.

CRC is computed over:

```text
Fixed Header + Payload
```

Recommended CRC:

```text
CRC16-CCITT
```

Purpose:

1. Detect corruption.
2. Reject malformed packets early.
3. Protect relay cache from polluted entries.

CRC is not a security mechanism.

---

# 10. MIC

Optional MIC provides authentication.

Recommended placement:

```text
Header + Payload + CRC + MIC
```

Recommended MVP MIC:

```text
4 bytes
```

Production recommendation:

```text
8 or 16 bytes
```

MIC should be computed over:

```text
Header + Payload + FrameCounter
```

MIC prevents:

- packet injection
- unauthorized commands
- fake ACK packets
- replay attacks when combined with counters

---

# 11. ACK Frame

ACK packets are minimal.

Recommended ACK payload:

| Field | Size |
| --- | --- |
| AckedSeq | 2 bytes |
| Status | 1 byte |

Optional fields:

| Field | Description |
| --- | --- |
| RSSI | Link quality |
| SNR | Link quality |
| Timestamp | Diagnostics |

ACK packets should be short to minimize airtime.

---

# 12. HELLO Frame

HELLO packets announce node existence.

Suggested HELLO payload:

| Field | Description |
| --- | --- |
| NodeType | endpoint / relay / base |
| FirmwareVersion | version |
| BatteryVoltage | optional |
| CapabilityFlags | optional |

HELLO packets are useful for:

- debugging
- deployment
- route learning
- topology visualization

HELLO traffic should be infrequent.

---

# 13. Future TLV Extension

Future LRP versions may support TLV extensions.

Suggested TLV structure:

```text
+------+--------+--------+
| Type | Length | Value  |
+------+--------+--------+
```

Advantages:

1. Backward-compatible extension.
2. Optional metadata.
3. Flexible diagnostics.
4. Route advertisement.
5. OTA metadata.

The MVP does not require TLV parsing.

---

# 14. Example Packet

Example telemetry packet:

```text
Version       = 0x01
Flags         = 0x01
PacketType    = 0x01
TTL           = 0x02
NetworkID     = 0x1001
SrcID         = 0x0101
DstID         = 0x0001
Seq           = 0x0042
PayloadLength = 0x08
Payload       = [temperature, humidity, battery]
CRC16         = 0xABCD
```

Semantic meaning:

```text
Node 0x0101 sends confirmed telemetry
through up to two relay hops
to base station 0x0001.
```

---

# 15. Duplicate Detection Key

Recommended deduplication key:

```text
NetworkID + SrcID + Seq + PacketType
```

Optional stronger dedup key:

```text
NetworkID + SrcID + DstID + Seq + PayloadHash
```

The relay cache should store:

| Field | Purpose |
| --- | --- |
| Dedup key | Duplicate detection |
| Expiration time | Cache cleanup |
| Forwarded flag | Optimization |
| RSSI/SNR | Optional diagnostics |

---

# 16. Fragmentation

The MVP should avoid fragmentation.

Large packets increase:

1. Airtime.
2. Collision probability.
3. Relay congestion.
4. Retransmission cost.

Future versions may support fragmentation with:

| Field | Purpose |
| --- | --- |
| FragmentID | Message identifier |
| FragmentIndex | Current fragment |
| FragmentCount | Total fragments |

Fragmentation should only be added if absolutely necessary.

---

# 17. Radio Independence

LRP frames are independent of radio implementation.

The same frame format may run over:

- SX1262
- SX1276
- LLCC68
- E22 UART modules
- RAK modules
- software simulation

The radio layer only transports opaque byte arrays.

---

# 18. Recommended Limits

| Parameter | Recommendation |
| --- | --- |
| TTL | ≤ 3 |
| Payload | ≤ 50 bytes |
| Retries | ≤ 3 |
| Confirmed traffic | Minimal |
| Relay count | Small |
| Broadcast usage | Rare |
| HELLO interval | 1–30 minutes |

---

# 19. MVP Compliance

A compliant MVP implementation must:

1. Parse fixed header.
2. Validate version.
3. Validate payload length.
4. Validate CRC.
5. Enforce TTL.
6. Maintain duplicate cache.
7. Support DATA packets.
8. Support ACK packets.
9. Forward eligible packets.
10. Drop invalid packets.
